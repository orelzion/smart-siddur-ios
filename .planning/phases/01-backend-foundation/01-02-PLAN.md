For each XML file, use the Read tool to read the file content, then manually extract `<string name="...">...</string>` entries from the XML text.

**2. Consolidates nusach variants:**

Group related strings into single prayer_content rows using the naming convention from MIGRATION_SPEC.md section 6.4:
- Base string (e.g., `Avot`) + variant strings (`AvotSefarad`, `AvotChabad`, `AvotAshkenaz`) -> ONE row with `text_variants: {edot: "...", sfarad: "...", chabad: "...", ashkenaz: "..."}`
- Detect variants by suffix pattern: strings ending in `Sefarad`, `Sfarad`, `Chabad`, `Ashkenaz`, `Edot` are variants of the base string
- Strings with no variants get `text_variants: {default: "..."}`
- The content_key follows dot-path convention derived from the source file and string name:
  - `18.xml` strings -> `amida.{name}` section
  - `shaharit.xml` strings -> `shacharit.{subsection}.{name}`
  - `arvit.xml` strings -> `arvit.{name}`
  - `mincha.xml` strings -> `mincha.{name}`
  - `mazon.xml` strings -> `mazon.{name}`
  - `musaf.xml` strings -> `musaf.{name}`
  - `tahanun.xml` strings -> `tahanun.{name}`
  - `globals.xml` strings -> `global.{name}`
  - `other.xml` strings -> `other.{name}`
  - `brachot.xml` strings -> `brachot.{name}`
  - `omer.xml` strings -> `omer.{name}`
  - `slichot.xml` strings -> `slihot.{name}`
  - `hagim.xml` strings -> `hagim.{name}`
  - `strings.xml` strings -> derive section from string name context or use `general.{name}`
  - Torah readings -> `torah.{book}.{name}`

**3. Converts placeholders:**

Replace all Android-style format strings with named placeholders per MIGRATION_SPEC.md section 2.4:
- `%1$s` -> context-dependent named placeholder (examine the generator that uses this string to determine the name)
- For bulk conversion where generator context is complex, use systematic naming: `{{insert_1}}`, `{{insert_2}}` etc. as interim names that can be refined later
- `klum` references (the empty-string pattern) -> `{{empty}}`
- Known mappings from MIGRATION_SPEC.md: `{{aseret_insert}}`, `{{winter_insert}}`, `{{summer_insert}}`, `{{anenu_insert}}`, `{{yaaleh_veyavo}}`, `{{al_hanissim}}`, `{{sick_name}}`, `{{pasuk}}`, `{{omer_count_hebrew}}`, `{{omer_count_translated}}`

**4. Parses translations from locale directories:**

Read translated strings from `values-iw/`, `values-fr/`, `values-es/`, `values-de/` and store in the `translations` JSONB column:
```json
{"en": "Amida", "he": "עמידה", "fr": "Amida", "es": "Amida", "de": "Amida"}
```
Note: The default values/ directory contains Hebrew prayer text (not English). The `values-iw/` contains Hebrew UI translations. Match translated strings by name attribute to base content rows.

**5. Parses JSON assets:**

- `slihot.json`: Structure is `{nusach: {section_number: [{id, title?, text, addToMenu?, expand?}]}}`. Each item references a string name in slichot.xml. Create prayer_content rows with content_key `slihot.{nusach}.{section}.{id}` and resolve the text references from the slichot.xml strings.
- `sukot.json`: Similar structure for Sukkot-specific prayers (hoshanot, ushpizin). Parse and create rows with content_key `sukot.{section}.{id}`.

**6. Inserts into prayer_content via Supabase:**

Use `mcp__supabase__execute_sql` to insert rows in batches of 50-100. For each row, set:
- `content_key`: dot-path key
- `content_type`: 'prayer' for prayer text, 'title' for section titles, 'instruction' for instructions, 'ui' for UI strings
- `text_variants`: JSONB with nusach keys
- `translations`: JSONB with locale keys (for title/UI strings)
- `placeholders`: TEXT[] listing named placeholders present in the text
- `section`: top-level section from content_key (e.g., 'amida', 'shacharit', 'arvit')
- `sort_order`: sequential within section
- `version`: 1 (initial)

**7. Creates initial content_versions entry:**

Insert a content_versions row recording the initial seed:
```sql
INSERT INTO content_versions (section, change_type, affected_keys, description)
VALUES ('all', 'create', ARRAY['initial_seed'], 'Initial content seeding from Android strings.xml and JSON assets');
```

**IMPORTANT NOTES:**
- Use INSERT ... ON CONFLICT (content_key) DO UPDATE for idempotency
- Handle HTML entities in XML strings (e.g., `&amp;`, `&lt;`, `\'`)
- Preserve HTML formatting tags in prayer text (`<br/>`, `<b>`, `<i>`, `<font color>`)
- Escape single quotes in SQL string literals (replace `'` with `''`)

<task type="auto">
  <name>Task 2: Import geo locations from cities1000.json</name>
  <files>supabase/seed/generate_geo_sql.py, supabase/seed/seed_geo_locations.ts</files>
  <action>
Import ~141K city records from `app/src/main/assets/cities1000.json` into the `geo_locations` table.

**Data mapping from cities1000.json to geo_locations:**

Each JSON entry has this structure (from inspection):
```json
{
  "geoname_id": "4931303",
  "name": "Bridgewater",
  "country_code": "US",
  "cou_name_en": "United States",
  "elevation": "30",
  "timezone": "America/New_York",
  "modification_date": "2017-05-23",
  "coordinates": {"lon": -70.97504, "lat": 41.99038}
}
```

Maps to geo_locations columns:
- `geoname_id` <- `geoname_id` (TEXT PK)
- `name` <- `name`
- `country_code` <- `country_code` (first 2 chars)
- `country_name` <- `cou_name_en` (field is named differently!)
- `elevation` <- `elevation` (TEXT, can be empty)
- `timezone` <- `timezone`
- `modification_date` <- `modification_date`
- `latitude` <- `coordinates.lat`
- `longitude` <- `coordinates.lon`

**Import strategy:**

The cities1000.json is 81MB -- too large to read in one go. The data is ALSO available in 4 split files of ~35K entries each:
- `app/src/main/assets/split_1.json` (35,248 entries)
- `app/src/main/assets/split_2.json` (35,248 entries)
- `app/src/main/assets/split_3.json` (35,248 entries)
- `app/src/main/assets/split_4.json` (35,248 entries)

**Execution method (single approach -- no alternatives):**

1. Create a Python3 script at `supabase/seed/generate_geo_sql.py` that:
   - Reads one split file (passed as CLI arg: `python3 generate_geo_sql.py split_1.json`)
   - Transforms each entry to match geo_locations schema (see mapping above)
   - Outputs SQL INSERT statements to stdout in batches of 500 rows
   - Each batch is one complete INSERT ... ON CONFLICT (geoname_id) DO NOTHING statement
   - Escapes single quotes in city/country names

2. For each split file, run the Python script via Bash to capture the SQL output:
   ```bash
   python3 supabase/seed/generate_geo_sql.py app/src/main/assets/split_1.json > /tmp/geo_batch.sql
   ```

3. Read the generated SQL file in chunks and execute each INSERT batch via `mcp__supabase__execute_sql`

4. Repeat for split_2.json, split_3.json, split_4.json

**Files created:**
- `supabase/seed/generate_geo_sql.py` -- Python3 script that reads a split JSON file and outputs SQL
- `supabase/seed/seed_geo_locations.ts` -- TypeScript documentation of the data mapping (NOT executed)
