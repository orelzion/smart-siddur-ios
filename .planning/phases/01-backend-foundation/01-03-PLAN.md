Create and deploy the `generate-prayer` Supabase Edge Function using `mcp__supabase__deploy_edge_function`.

**Request/Response contracts** (from MIGRATION_SPEC.md section 4.2):

Input: `GeneratePrayerRequest` with prayer_type, date (ISO 8601), nusach, tfila_mode, location, settings.
Output: `GeneratePrayerResponse` with items[], menu[], metadata.

**Architecture:**

The edge function should be structured as:

```
index.ts
  - Deno.serve handler
  - Parse and validate request
  - Initialize JewishCalendar from kosher-zmanim for the given date
  - Determine calendar state (isShabbat, isYomTov, isRoshChodesh, isTaanis, isAseret, etc.)
  - Determine seasonal state (isWinter for Mashiv HaRuach, isTalUmatar, etc.)
  - Call the appropriate prayer assembler based on prayer_type
  - Each assembler queries prayer_content for its section's rows
  - Resolve text_variants by nusach (try request nusach key, fallback to "default")
  - Resolve {{named}} placeholders with actual values
  - Filter by tfila_mode (YAHID removes kedusha/kadish/barchu)
  - Apply user state filters (is_avel, no_tahanun, is_vanenu, is_woman)
  - Build menu entries from items with add_to_menu=true
  - Return response
```

**Key implementation details:**

1. **kosher-zmanim integration**: Import from `npm:kosher-zmanim` (available in Deno). Use `JewishCalendar` class to determine:
   - `jewishMonth`, `jewishDayOfMonth`, `jewishYear`
   - `isRoshChodesh()`, `isChanukah()`, `isPurim()`, `isYomTov()`, `isCholHamoed()`
   - `isRoshHashana()`, `isYomKippur()`, `isSukkos()`, `isPesach()`, `isShavuos()`
   - `isTaanis()` (fast days)
   - Day of week for Shabbat detection
   - Omer count

2. **Calendar state derivation** (port from Zemanim.java):
   - `isAseret()`: Between Rosh Hashana and Yom Kippur (inclusive)
   - `isWinter()`: From Shmini Atzeret (22 Tishrei) to Pesach (15 Nisan) -- for Mashiv HaRuach
   - `isTalUmatar()`: Israel: from 7 Cheshvan. Diaspora: from Dec 4/5 (depends on civil year). Until Pesach.
   - `isSummer()`: Not winter -- for Morid HaTal
   - Shabbat: day of week === 7 (Saturday) in Jewish calendar context
   - Erev Shabbat: day of week === 6 (Friday)

3. **Nusach resolution**: For each prayer_content row:
   ```typescript
   const text = row.text_variants[request.nusach] || row.text_variants['default'] || '';
   ```

4. **Placeholder resolution**: Replace `{{name}}` patterns:
   - `{{empty}}` -> `""`
   - `{{sick_name}}` -> `request.settings.sick_name`
   - `{{pasuk}}` -> `request.settings.pasuk`
   - `{{aseret_insert}}` -> fetch text from `amida.avot_aseret_insert` if isAseret, else ""
   - `{{winter_insert}}` -> fetch text from `amida.gevurot_winter_insert` if isWinter, else ""
   - `{{summer_insert}}` -> fetch text from `amida.gevurot_summer_insert` if isSummer, else ""
   - `{{yaaleh_veyavo}}` -> appropriate text if Rosh Chodesh/Yom Tov/Chol HaMoed
   - `{{al_hanissim}}` -> appropriate text if Chanukah/Purim
   - `{{anenu_insert}}` -> appropriate text if fast day and settings.is_vanenu
   - `{{omer_count_hebrew}}` -> calculated from JewishCalendar omer day
   - `{{ref:content_key}}` -> fetch and inline text from referenced prayer_content row

5. **Tfila mode filtering** (from Android generators):
   - `YAHID` (praying alone): Remove items with content_key containing 'kedusha', 'kadish', 'barchu'. These are minyan-only prayers.
   - `CHAZAN` (leading prayer): Include all items, some may get extra content (e.g., full repetition)
   - `REGULAR`: Standard set

6. **User state filtering**:
   - `is_avel` (mourner): Skip hallel, skip certain psalms
   - `no_tahanun`: Skip tahanun section entirely
   - `is_vanenu`: Include anenu on fast days
   - `is_woman`: May affect certain brachos wording
   - `nachem_always`: Include nachem on Tisha B'Av

7. **Prayer type assemblers**: Each prayer type needs its own assembly logic. The major ones:
   - `shacharit`: birkot_hashachar -> zimra -> shma -> amida -> tahanun -> post (most complex, delegates to sub-generators)
   - `mincha`: ashrei -> amida -> tahanun (for mincha, uses different tahanun rules)
   - `arvit`: shma -> amida (no tahanun at night)
   - `mazon`: multiple variants (regular, guest, wedding, brit_mila)
   - `omer`: count text with date-specific number
   - All 26+ types listed in PrayerType enum

For this initial implementation, focus on getting the architecture right with full support for:
- Shacharit (most complex, proves the architecture)
- Mincha
- Arvit
- Mazon
- Omer
- The "other" simpler types (al_mita, haderech, havdala, hanuka, levana, blessings, etc.)

The remaining complex types (kinot, slihot, torah_reading, musaf) can return their content in a simpler format since their assembly logic is more straightforward (sequential content sections).

Create and deploy the `generate-prayer-batch` Supabase Edge Function.

**Request contract:**
```typescript
interface GeneratePrayerBatchRequest {
  prayer_types: PrayerType[];       // e.g., ['shacharit', 'mincha', 'arvit']
  start_date: string;               // ISO 8601: '2026-02-08'
  days: number;                     // typically 14
  nusach: 'edot' | 'sfarad' | 'ashkenaz' | 'chabad';
  tfila_mode: 'regular' | 'yahid' | 'chazan';
  location: { latitude, longitude, elevation, timezone_id, country_code, is_in_israel };
  settings: { is_woman, is_avel, no_tahanun, is_vanenu, nachem_always, tal_preference, is_mizrochnik, mukaf_mode, sick_name, pasuk, language };
}
```

**Response contract:**
```typescript
interface GeneratePrayerBatchResponse {
  prayers: Array<{
    date: string;
    prayer_type: string;
    items: PrayerTextItem[];
    menu: MenuEntry[];
    metadata: PrayerMetadata;
  }>;
  generated_at: string;
  content_version: number;
}
```

**Implementation:**

The batch function should:
1. Parse and validate the request
2. Pre-fetch ALL prayer_content rows needed (one query to get all rows, cache in memory for the batch)
3. Loop through each date in range (start_date to start_date + days - 1):
   a. Create JewishCalendar for the date
   b. For each prayer_type in the request:
      - Call the same prayer assembly logic as generate-prayer
      - Collect the result
4. Return all results in one response

**Optimization:**
- Pre-fetch prayer_content once, not per prayer. This is the main performance win of the batch endpoint.
- The JewishCalendar must be re-created per date (calendar state changes daily).
- Share the assembly logic between generate-prayer and generate-prayer-batch. The recommended approach is to import shared modules. However, since Edge Functions are deployed independently, the shared logic should be duplicated in the batch function or use Deno's `import` from a shared URL.

**Practical approach for shared code:**
Since both functions need the same prayer assembly logic, structure the code as:
- A shared module with all the assembly logic (calendar helpers, placeholder resolution, prayer assemblers)
- Both edge functions import this shared module

For Supabase Edge Functions, shared code can be placed in `supabase/functions/_shared/` directory and imported relatively.

Deploy using `mcp__supabase__deploy_edge_function` with:
- name: "generate-prayer-batch"
- verify_jwt: true
- Include the shared code as additional files

**Performance guard:**
- Limit `days` to max 30 (return 400 if exceeded)
- Limit `prayer_types` to max 10 per request
- Total generated prayers per request: types * days <= 100
